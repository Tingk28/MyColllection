Computational Thinking and Problem Solving

Problem Definition

  • Our target problem -
    怎麼樣才能減少助教批改作業的時間

Problem Decomposition

為什麼助教批改作業要花這麼多時間

  • 出作業的方式哪裡有問題？
    • (1)作業成果格式不一致，有圖片有文字，很累，降低效率
  • 改作業的方式哪裡有問題？
    • (2)學生會遲交作業，所以需要不斷檢查有沒有人補交
    • (3)批改作業時要將程式碼複製下來執行，需要時間
    • (4)FB名單與選課系統不一致，對照需要時間
  • 交作業的方式哪裡有問題？
    • (5)尋找每篇作業所在的貼文需要時間
    • (6)沒有可以統計作業繳交狀況 的方法，人工統計浪費時間
    • (7)繳交CodiMD網址，還要再點進去，麻煩
  • 助教人數哪裡有問題？
    • (8)助教人太少，批改速度太慢

Pattern Recognition

  • 出作業

    • (1)出作業時是按照當週Udacity內容而定，缺乏格式

  • 改作業

    • (2)助教沒辦法統計哪些作業是新的、或需要之後批改的
    • (3)批改時要不斷執行複製貼上的動作，所花時間太久

  • 交作業

    • (4)學生FB的名字不一定是自己的中文姓名
    • (5)(6)學生繳交作業的位置混亂，沒有一個能一目瞭然的表單供助教查詢
    • (7)目前都是用CodiMD繳作業，再貼到FB上
    • (8)大家都擠在最後一天交
    • (8)改太多作業會不快樂，降低效率
    • (8)助教不想改作業

Abstraction

  • 出作業
    • (1)作業的內容標準化
  • 交/改作業
    • (2)自動統計助教批改過的作業
    • (3)自動執行或批改
    • (4)自動轉譯學生FB名稱
    • (5)(6)自動尋找學生交的作業位置
    • (7)自動打開CodiMD
    • (8)自動新增助教(o

Algorithm

Subproblem:

  1. 作業的內容標準化
    • 定義未來的作業內容 = 標準化輸入輸出
  2. 自動統計助教批改過的作業
    • 從資料庫存/讀紀錄
  3. 自動執行或批改
    • 解決方案選項
      • 用爬蟲爬下程式碼自動執行
        • 好
      • 用colab直接在上面跑
        • 更好
      • 在網站上自動執行並批改，完全不需要助教(參考Udacity)
        • 助教一開始很累
    • 最後抉擇
      • 將抓到的程式碼內容儲存成python檔並執行
      • 將結果顯示在 UI 上
  4. 自動轉譯學生FB名稱
    • 從助教那邊取得學生的 FB 名稱清單
    • 將名稱清單儲存成csv並在讀取時自動轉換
  5. 自動尋找學生交的作業位置
    • 解決方案選項
      • 讓電腦自動執行程式碼並輸出
        • 作業尚無標準化之前無法達到
      • 用Google表單交作業
        • 學生無法看自己繳交了沒
        • 助教那邊不會收到繳交通知
        • 同學不能針對特定作業問問題
      • 用Google試算表交作業
        • 助教那邊不會收到繳交通知
        • 沒有時間戳記，有人會偷偷遲交
      • 用Colab交作業
        • 很棒，但我們不知道（，請等待2.0版
      • 用FTP交作業
        • 助教那邊不會收到繳交通知
        • 助教不能給回覆
        • 同學不能針對特定作業問問題
      • 用Moodle交作業
        • 學生不能左右觀察法
      • 助教自己架一個網站給學生交作業
        • 助教會先苦一陣（學）子（期）(需要兩代人的努力)
      • 缺助教
        • 沒有2倍薪解決不了的事，如果有，那就4倍
    • 最後抉擇
      • 用爬蟲登入、搜尋、找到繳交作業連結頁面
  6. 自動打開 CodiMD 並取得內容
    • 用爬蟲

達成的事項

  • 使用PyQt整合UI，提供統一操作介面
    • 登入FB帳號
    • 選擇欲批改的作業
    • 所有作業的批改狀態一目了然
    • 快速切換上一份/下一份作業
  • 自動抓取CodiMD程式碼區段並執行
    • 使用者僅需選擇欲批改的作業
    • 選擇到該作業時便已執行完程式碼區段
    • 助教會直接看到執行結果及原始碼
  • 可以紀錄上次批改到哪位同學的作業
    • 方便記錄工作狀態，不會忘記改過哪些作業
  • 可以快速且清楚的標示或開啟已批改、未批改跟批改未過的作業
    • 提供3種狀態：已通過、未通過、未批改

同場加映: 怎麼把上面的部分進行分工

問題:

如何有效率的分工 = 如何把循序開發的舊模式平行化

拆解:

程式每一階段的輸出會接到下一階段的輸入
在程式開始寫之前不知道會需要什麼輸入

pattern:

我們寫程式都是想到哪寫到哪

抽象化:

找出整個程式每一階段會需要的資料
把需要相同資料的部分集合成同個 subproblem
不同群體分給不同人

演算法:

用記事本寫下 pseudo code
把 pseudo code 中資料的來源區段標記出來
取得區段與區段的關係圖
把不相關的區段分給不同人做
最後需要有人把所有區段整合起來
